1. What was fixed in this session

Three defects were diagnosed and fixed. All changes are in the working tree.

① “Restore version” did nothing / reverted on refresh FIXED

Restore pushed content into the editor view with editor.commands.setContent(). Because the document is owned by the Yjs content fragment (TipTap Collaboration), the binding reverted it and nothing persisted.

Before
editor.commands.setContent(json) — view only; lost on sync/refresh.
After
A real ProseMirror transaction (tr.replaceWith) the Yjs sync plugin observes → writes into the shared doc → propagates to collaborators and persists.

File: src/components/version-history/VersionPanel.tsx — satisfies “Restore previous versions / Safe restoration / Never corrupt active sessions”.

② Realtime sync stuck in an infinite reconnect loop FIXED

The browser looped on WS connection error, fetching new token and reconnecting…. Realtime sync uses a separate WebSocket relay (relay/src/server.ts, port 1234), but npm run dev only started Next.js — nothing listened on 1234, so the client retried every 2s forever. “Syncing” was the retry state, not a live connection.

Before
dev = next dev only. Relay never launched; no scripts to run it.
After
Added relay scripts + a zero-dependency launcher so npm run dev runs both Next.js and the relay.

Files: relay/package.json (new dev/build/start), scripts/dev.mjs (launcher), package.json (dev, dev:next, dev:relay). Verified: relay boots and answers HTTP 200 on :1234.

③ Duplicate undo/redo system (TipTap warning) FIXED

StarterKit.configure({ history:false }) was silently ignored — in TipTap v3 the option was renamed to undoRedo, so StarterKit's undo stack ran alongside Collaboration's, causing the “not compatible with @tiptap/extension-undo-redo” warning. Changed to undoRedo:false.

File: src/components/editor/Editor.tsx.

3. Architecture (as built)

Data & sync flow

  • Typing → Yjs CRDT → IndexedDB (instant, offline).
  • Dexie outbox queues updates; sync engine pushes/pulls the durable log in Postgres with retry.
  • WebSocket relay fans out live updates between collaborators (JWT-gated, role-aware).
  • Snapshots store full encoded state for version history & restore.

Two processes to run/deploy

  • Next.js app (Vercel) — UI, route handlers, auth, AI, durable-log + snapshot APIs, WS token minting.
  • Relay (long-lived Node service) — y-websocket rooms; cannot run on Vercel serverless.
  • PostgreSQL (Neon/Supabase) — users, documents, members, crdt_updates, snapshots, with RLS.

4.2 Deploy steps

# 1) Database
npx prisma migrate deploy        # applies schema + RLS policies

# 2) Next.js app on Vercel
# set all App env vars in the Vercel dashboard (never commit .env)
vercel deploy --prod             # build runs: prisma generate && next build

# 3) Relay on a long-lived host (Railway / Fly.io / Render)
cd relay && npm ci && npm run build && npm start
# expose it over wss:// and point NEXT_PUBLIC_WS_URL at it

Local dev now needs only npm run dev — it starts Next.js and the relay together.

4.3 Pre-launch checklist

  • ☐ Rotate every secret in .env (it currently holds real OAuth/DB/AI creds) and remove it from git history.
  • ☐ Replace the weak AUTH_SECRET (my-super-secret-…) with a 32-byte random value in both app & relay.
  • ☐ Set NEXT_PUBLIC_WS_URL to the deployed relay; verify two browsers sync live.
  • ☐ Deploy the relay as its own service; add a /health check + restart policy.
  • ☐ Confirm OAuth redirect URIs point to the production domain.
  • ☐ Add the submission footer (Name · GitHub · LinkedIn).
  • ☐ Swap in Redis-backed rate limiting if running more than one app instance.
